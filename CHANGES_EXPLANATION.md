# Внесённые изменения в проект CarDealershipGraph

## 1. Изменение атрибута FuelConsumptionCity → FuelConsumption
**Где:** Таблица `Models`, строка 41  
**Было:** `FuelConsumptionCity DECIMAL(4,1)`  
**Стало:** `FuelConsumption DECIMAL(4,1)`  
**Причина:** Атрибут переименован для более общего названия, так как значение может представлять расход топлива не только в городе, но и в смешанном цикле или других условиях.

---

## 2. Разделение имени клиента на два атрибута
**Где:** Таблица `Customers`, строки 67-68  
**Было:** `CustomerName NVARCHAR(100) NOT NULL`  
**Стало:** 
- `CustomerFirstName NVARCHAR(100) NOT NULL`
- `CustomerSecondName NVARCHAR(100)`  

**Причина:** Разделение имени и фамилии позволяет:
- Выполнять поиск по фамилии отдельно
- Корректно сортировать клиентов по фамилии
- Соответствует лучшим практикам проектирования БД
- Упрощает персонализацию обращений (например, "Уважаемый Иван" вместо "Уважаемый Иван Петров")

**Затронутые запросы:** Все SELECT-запросы, использующие имя клиента, теперь используют `CONCAT(CustomerFirstName, ' ', CustomerSecondName)`.

---

## 3. Изменение комментария таблицы BELONGS_TO
**Где:** Строка 79  
**Было:** `-- РЕБРО 1: BELONGS_TO (Модель принадлежит Марке)`  
**Стало:** `-- РЕБРО 1: BELONGS_TO (Модель выпускается Маркой)`  
**Причина:** Семантическое уточнение смысла связи — модель не "принадлежит" марке как собственность, а производится (выпускается) этой маркой.

---

## 4. Переименование таблицы SERVES_BRAND → SERVES
**Где:** Строки 96-113 и все упоминания в запросах  
**Было:** `SERVES_BRAND`  
**Стало:** `SERVES`  
**Причина:** Более краткое и универсальное название. Смысл связи остаётся тем же — сервисный центр обслуживает марки автомобилей.

## 5. Ответ на вопрос о дублировании атрибутов даты

### Вопрос:
В таблице `BELONGS_TO` есть атрибуты `StartDate DATE`, `EndDate DATE`.  
В таблице `Models` есть атрибуты `ProductionStartYear SMALLINT`, `ProductionEndYear SMALLINT`.  
**Нужно ли это повторение?**

### Ответ:
**Да, это оправданное дублирование с разными целями:**

| Атрибут | Где находится | Семантика | Пример использования |
|---------|--------------|-----------|---------------------|
| `ProductionStartYear` / `ProductionEndYear` | `Models` (узел) | **Технические данные модели** — когда эта модель вообще начала/прекратила выпускаться компанией | Фильтрация всех моделей по году выпуска, аналитика по поколениям моделей |
| `StartDate` / `EndDate` | `BELONGS_TO` (ребро) | **История принадлежности** — когда конкретная модель начала/перестала относиться к конкретной марке | Отслеживание рестайлингов, смены поколений, случаев когда модель передавалась между брендами внутри концерна |

### Практический пример:
Модель **Opel Vectra** производилась с 1988 по 2008 год (`ProductionStartYear=1988`, `ProductionEndYear=2008` в узле Models).

Но если бы она меняла бренд (гипотетически):
- 1988-1995: принадлежала Opel (ребро BELONGS_TO с StartDate='1988-01-01', EndDate='1995-12-31')
- 1995-2008: принадлежала Vauxhall (ребро BELONGS_TO с StartDate='1996-01-01', EndDate='2008-12-31')

**Вывод:** Атрибуты в узле хранят **общий жизненный цикл модели**, а атрибуты в ребре хранят **историю связи с конкретной маркой**. Это не избыточность, а нормализация для поддержки временных графовых запросов.

## 6. Объяснение выбора ON DELETE для таблиц-рёбер

### BELONGS_TO: `ON DELETE NO ACTION`
**Связь:** Models → Brands

**Почему NO ACTION:**
- Если удалить марку (Brand), нельзя автоматически удалить все модели этой марки — это потеря важных исторических данных
- Если удалить модель, связь должна остаться для истории (модель могла быть перепродана, обслуживаться и т.д.)
- Требуется явное подтверждение от пользователя перед удалением связанных записей
- **Бизнес-логика:** Марки могут исчезать с рынка, но информация о выпущенных моделях должна сохраняться

### SERVES: `ON DELETE NO ACTION`
**Связь:** ServiceCenters → Brands

**Почему NO ACTION:**
- При удалении бренда нельзя терять информацию о том, какие сервисные центры его обслуживали — это история партнёрства
- При удалении сервисного центра нужно сохранить историю какие бренды он обслуживал
- Контракты между сервисами и брендами имеют юридическую значимость, их история должна сохраняться
- **Бизнес-логика:** История отношений с сервисными центрами важна для аудита и анализа

### PURCHASED_BY: `ON DELETE CASCADE`
**Связь:** Customers → Models

**Почему CASCADE:**
- При удалении клиента все записи о его покупках должны удаляться автоматически (требование GDPR/законов о персональных данных)
- При удалении модели из базы (редкий случай) записи о её покупках теряют смысл
- **Бизнес-логика:** Покупка не существует без покупателя — это зависимая сущность
- Упрощает удаление тестовых данных и коррекцию ошибок ввода

### Сводная таблица:

| Таблица-ребро | ON DELETE | Обоснование |
|---------------|-----------|-------------|
| BELONGS_TO | NO ACTION | Сохранение истории производства моделей |
| SERVES | NO ACTION | Сохранение истории партнёрских отношений |
| PURCHASED_BY | CASCADE | Зависимость записи о покупке от существования клиента |

## Итог
Все изменения направлены на:
1. Улучшение семантики модели данных
2. Соблюдение лучших практик проектирования БД
3. Поддержку исторических данных там, где это важно для бизнеса
4. Защиту целостности данных через правильно выбранные ограничения целостности
